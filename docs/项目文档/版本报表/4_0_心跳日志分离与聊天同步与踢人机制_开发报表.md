# GdxCore v0.9.3 心跳日志分离 & PublicIPResolver 无头修复 & 聊天网络行为 & 踢人/断连机制 — 开发报表

## 一、概述

本次开发围绕 **5 个问题/需求** 展开，涉及日志过滤、网络诊断、聊天系统设计、玩家管理和传输层断连：

1. **PhoenixChannel 心跳日志分离** — 将 PhoenixChannel 日志拆分为 3 个独立 TAG（生命周期 / 心跳 / 线级），心跳和线级默认黑名单拦截。
2. **PublicIPResolver 无头环境修复** — 添加 Java 原生 HTTP 后备路径，解决 `HeadlessApplication` 中 `Gdx.net` 可能静默失败的问题。
3. **ChatChannel 网络发送行为变更** — `processInput()` 在有 `networkSender` 时仅调用 sender（不再本地 post），允许 Host/Client/Server 各自定义发送流程。
4. **HeadlessGameServer 玩家管理增强** — 新增 `getPlayerNames()` / `onBeforeKick()` / 完整 `kickPlayer()` 四步流程。
5. **传输层 disconnectClient** — `UdpSocketTransport` 和 `ReliableUdpTransport` 新增主动断连客户端能力。

## 三、架构设计

### 3.1 PhoenixChannel 日志 TAG 拆分

```
TAG = "PhoenixChannel"    ← 连接/加入/离开/错误等生命周期事件
TAG_HEARTBEAT = "PhoenixHB"   ← 心跳发送/回复（默认黑名单拦截）
TAG_WIRE = "PhoenixWire"       ← >>> / <<< 线级收发日志（默认黑名单拦截）
```

**设计动机**：之前所有 PhoenixChannel 日志共用一个 TAG，心跳和线级消息频率极高（每 30s 心跳 + 每次 presence/broadcast 的收发），淹没了真正有用的生命周期事件。拆分后可独立控制过滤级别。

### 3.2 PublicIPResolver 双路径架构

```
resolvePublicIP(callback)
    │
    ├─ 首选: Gdx.net.sendHttpRequest()
    │       ├─ 成功 → callback(ip)
    │       └─ 失败/取消/异常 → 自动 fallback ↓
    │
    └─ 后备: resolvePublicIPNative()
              ├─ new Thread("PublicIPResolver")
              │   └─ HttpURLConnection → api.ipify.org
              ├─ 成功 → postRunnable(callback(ip))  (Gdx.app 可用时)
              │         callback(ip)                  (无头环境)
              └─ 失败 → callback(null)
```

**设计动机**：`HeadlessApplication` 的 `Gdx.net` 实现可能不完整或静默失败，导致服务器永远获取不到公网 IP，进而无法在大厅注册房间。原生 Java HTTP 作为后备路径确保在任何 JVM 环境下都能工作。

### 3.3 ChatChannel 网络发送行为

```
processInput(text)
    │
    ├─ networkSender != null:
    │   └─ networkSender.sendChat(msg)  ← 仅此一步，不本地 post
    │       （由 Host/Client/Server 各自决定是否 post + broadcast）
    │
    └─ networkSender == null:
        └─ postMessage(msg)  ← 纯本地模式，直接 post
```

**非兼容变更说明**：之前 `processInput()` 无论是否有 networkSender 都会先本地 post，再调用 sender。这导致 Host 模式会出现消息重复（本地 post 一次 + broadcast 时又 post 一次）。新行为要求调用方在 networkSender 实现中自行决定本地 post 时机。

### 3.4 kickPlayer 四步流程

```
kickPlayer(clientId)
    │
    ├─ 1. onBeforeKick(clientId)        ← 子类 hook：发送 RPC 通知等
    ├─ 2. manager.despawnByOwner(clientId)  ← 销毁该客户端所有 NetworkObject
    ├─ 3. onPlayerDisconnected(clientId)    ← 触发业务层离线回调
    └─ 4. transport.disconnectClient(clientId)  ← 传输层断开 + 更新大厅人数
```

**设计动机**：之前 `kickPlayer()` 只做了 despawn，没有断开传输层连接，也没有通知客户端。客户端被踢后仍保持 UDP 连接，无法感知被踢状态。新流程确保：RPC 通知先于 despawn 发送（否则 NetworkBehaviour 已销毁无法发 RPC）→ 清理对象 → 回调业务 → 传输层断开。

### 3.5 Transport disconnectClient

**UdpSocketTransport**:
- 发送 `DISCONNECT_MAGIC` 3 次（UDP 不可靠，多发保证送达概率）
- 从 `activeClientIds` 移除

**ReliableUdpTransport**:
- 委托 rawTransport.disconnectClient()
- 额外清理 `clientReceiveTrackers` 和 `clientLastRecvTimeMs`

## 四、非兼容变更 (Breaking Changes)

| 变更 | 影响 | 迁移方式 |
|------|------|----------|
| `ChatChannel.processInput()` 不再自动本地 post | 使用 networkSender 的代码需自行 post | 在 networkSender lambda 中添加 `ChatChannel.get().postMessage()` |
| `HeadlessGameServer.kickPlayer()` 签名不变但行为变更 | 子类如果 override 了 kickPlayer 需要适配 | 改为 override `onBeforeKick()` hook |
| `HeadlessGameServer` 新增 `getPlayerNames()` | 子类可选 override | 默认返回 "Player#id" |
| Transport 新增 `disconnectClient()` | 自定义 Transport 实现需适配 | 实现该方法 |
